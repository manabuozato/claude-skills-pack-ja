# Plaud文字起こし取り込みスキル

## 概要

Plaudからエクスポートした文字起こしtxtファイルを読み取り、NotionのミーティングDBにページとして作成する。
Zapierによる自動登録（Zoom録画→DB登録）が併走しているため、重複チェックが必須。

## 固定ID

| 対象 | ID |
|------|-----|
| ミーティングDB データソース | collection://77760381-fd3e-49e5-b5c3-d24418468b4f |
| Tactiq受け箱（フォールバック先） | 3001bdf3080981dca3f6f08510bc8b23 |

## 文体ルール

ダッシュ系つなぎ禁止。太字強調禁止。

## Plaud txtファイルの構造

ファイル名パターン: `MM-DD_ミーティング名-transcript__N_.txt`
- MM-DD: 月日（年はファイルのコンテキストから推定、なければ当年）
- ミーティング名: アンダースコア区切り。濁点等がカタカナ分離されている場合あり
- N: トランスクリプト番号（通常1）

本文構造:
```
HH:MM:SS Speaker N
発言テキスト
HH:MM:SS Speaker N
発言テキスト
...
```

特徴:
- 話者は「Speaker 1」「Speaker 2」等の匿名表記（Tactiqと異なり実名なし）
- タイムスタンプは HH:MM:SS 形式
- 発言は複数行にわたることがある

## 実行手順

### Step 0: モデル確認
- Opusで実行されている場合、ユーザーに通知する:「現在Opusで実行中です。Plaud取り込みはSonnetで十分対応可能です。モデル切替を検討してください。」
- 通知後、ユーザーの指示に従う（続行 or 中断）

### Step 1: ファイル解析

アップロードされたtxtファイルから以下を抽出:
- ファイル名から日付（MM-DD → YYYY-MM-DD。年が不明なら当年を仮定）
- ファイル名からミーティング名（アンダースコアをスペースに置換、カタカナ濁点分離を修復、-transcript以降を除去）
- 本文から話者数（Speaker Nの最大値）
- 本文から開始時刻（最初のタイムスタンプ）と終了時刻（最後のタイムスタンプ）
- 本文冒頭から議題・要点のヒントを抽出（最初の数百文字程度を読む）

### Step 2: 重複チェック（Zapier併走対応）

DB登録前に必ず重複チェックを行う。

#### チェック手順
1. ミーティングDB（data_source: `collection://77760381-fd3e-49e5-b5c3-d24418468b4f`）をNotion searchで検索
2. 検索条件: Step 1で特定した日付の前後1日 + ミーティング名のキーワード
3. 同プロジェクトで同日のエントリがないか確認

#### 重複パターンと対応

| パターン | 状況 | 対応 |
|---|---|---|
| 完全一致 | 同日・同タイトルのPlaud登録が既にある | ユーザーに確認（二重取り込み防止） |
| Zapier版あり | 同日に録音手段「Zoom」のエントリがある | ユーザーに選択肢提示（下記参照） |
| 類似タイトル | 同日に似たタイトルのエントリがある | ユーザーに確認（別ミーティングかどうか） |
| 該当なし | 重複なし | そのまま新規登録 |

#### Zapier版が存在する場合の選択肢
ユーザーに以下を提示:
- 統合: Zapier版のページにPlaud文字起こしを追記（推奨。Zapier版の既存プロパティを維持しつつ、Plaud文字起こしをページ本文に追加）
- 上書き: Zapier版を削除してPlaud版で新規登録
- 並存: 別エントリとして両方残す（録音手段で区別）
- スキップ: 登録しない

統合の場合の処理:
1. Zapier版のページIDを取得
2. notion-update-pageのupdate_contentで、ページ本文末尾にPlaud文字起こしを追加
3. 録音手段プロパティを「Zoom」→ 変更しない（Zapier版のメタデータを尊重）
4. 要約が空なら生成して追加

### Step 3: ミーティングDBにページ作成（新規登録の場合）

Notion create-pages で、ミーティングDBデータソースに新規ページを作成。

プロパティ:
- ミーティング名: Step 1で抽出したタイトル（クリーニング済み）
- date:日時:start: YYYY-MM-DDThh:mm:ss（開始時刻。ファイル名の日付+本文の最初のタイムスタンプ）
- date:日時:is_datetime: 1
- プロジェクト: タイトルやキーワードから判定（下記ルール参照）
- ステータス: 完了
- 録音手段: Plaud
- 参加者: Speaker数から「○名参加」と記載。ユーザーが補足すれば更新。
- 要約: Claude生成の要約（3〜5文。下記ガイドライン参照）
- 議題: ファイル名のミーティング名部分から抽出
- アクションアイテム: 文字起こしから検出できたもの（なければ空欄）

ページ本文: 以下の構成で記載。

```
## 文字起こし（Plaud）

（文字起こし全文を貼り付け）
```

長すぎる場合（Notion APIの制限）はトリムして「（以下省略、元ファイル参照）」と付記。

### Step 4: ユーザーへの報告

作成/統合完了後、以下を報告:
- 実行した処理（新規作成 / Zapier版に統合 / 上書き / 並存）
- 作成/更新したページのタイトルとURL
- 埋めたプロパティの一覧
- 「参加者名やプロジェクト分類を修正したい場合はお知らせください」と案内

## プロジェクト判定ルール

ファイル名やミーティング名から以下のキーワードで判定:
- CFF, Creativity Future Forum, クリエイティビティフューチャーフォーラム → CFF
- CIRE, シェフ, レジデンス, Chef → CIRE
- 研究, 由井薗, JAIST, ジャイスト, 嶋田, 論文, 博士 → 研究
- UoC, ゼミ, Gov, ガバナンス, マンガ, 漫画 → UoC
- 日置, 働き方PT → 日置市
- 都政新報 → 都政新報
- 博報堂, 勝野 → 博報堂
- 上記に該当しない → その他

## 要約生成ガイドライン
DB「要約」プロパティに入れる要約は以下の基準で生成する:
- 3〜5文の簡潔な日本語
- 構成: 主要議題 → 決定事項/合意事項 → 次のアクション/懸念事項
- Plaudの文字起こしはSpeaker表記で匿名のため、内容ベースで要約する
- 文字起こし品質が低い場合は「（※音声品質により要約精度が低い可能性あり）」と付記

## 複数ファイル対応

複数のtxtが同時にアップロードされた場合、1件ずつ順番に処理する。各ファイルごとに重複チェックを実施。

## 注意事項

- Plaudの文字起こしはSpeaker N表記で匿名のため、参加者名は自動では埋められない
- ファイル名の濁点分離（例: 「ク_」が「グ」の意味）は文脈から修復を試みるが、不確実な場合はそのまま残す
- 年の推定: ファイル内に年情報がない場合、当年を仮定する。内容から明らかに異なる場合はユーザーに確認
- Zapierが同じミーティングを先に登録している可能性が常にある。重複チェック（Step 2）をスキップしない
